# Идеи модели удержания клиента в рамках GeoATM (VTB / HSE)

Контекст проекта: агрегаты по ячейкам H3 (спрос, клиенты, суммы, POI, конкуренты), эвристический Demand Score, ML-приоритет, сценарные теги, новостройки с `delivery_score`. Ниже — направления, которые **реалистично стыкуются** с текущими данными и картой, без обещания «полной CRM» на одном CSV.

## 1. Индекс «лояльности зоны» (proxy retention)

- **Смысл:** не предсказать отток по ФИО, а оценить **стабильность и «липкость»** спроса в ячейке: относительно медианы по городу смотреть `sum_per_customer`, `transactions_per_customer`, волатильность (`avg_std` как прокси), долю повторных визитов, если появятся помесячные ряды.
- **Зачем:** отличать «разовый трафик» от зоны, куда клиенты **возвращаются** — туда удержание через точку входа (АТМ/офис) имеет смысл.
- **Реализация:** новый столбец `retention_proxy_score` (0–1) из нормированных признаков + правило в `features.py` и строка в попапе/API.

## 2. «Домашняя» vs «транзитная» зона

- **Смысл:** зона с высоким POI (метро, ТЦ, ВУЗ) и высокой плотностью конкурентов может давать **оборот без удержания**; зона с жилой застройкой и стабильным `unique_customers` — кандидат на **удержание через сервис** (кэшбэк, пакеты, курьерские точки).
- **Реализация:** простые правила по порогам `metro_count`, `mall_count`, `competitor_atm_count`, `vtb_atm_count` + тег `transit_heavy` / `residential_like` (при появлении жилой застройки из внешних данных).

## 3. Конкурентное давление и «окно» для удержания

- Уже есть **ближайший конкурент и число АТМ в радиусе**. Можно ввести **pressure_score**: чем ближе сильный конкурент и чем ниже уникальность клиента ВТБ в ячейке, тем выше риск перетекания.
- **Удержание:** подсказки в UI «рядом агрессивный конкурент — приоритет оффера/комиссии на снятие в этой зоне» (без автоматизации продуктов — только аналитика для бизнеса).

## 4. Связка с новостройками (`delivery_score`)

- Новые жители = **новые привычки к банку**. Зоны рядом с ЖК с высоким `delivery_score` и низким `vtb_atm_count` — **окно первых 6–12 месяцев** для кампаний удержания (карта уже показывает точки).
- **Реализация:** склеить H3 зоны с буфером вокруг `new_buildings` (пространственный join), метка `new_residents_inflow` в зоне.

## 5. Сегментация сценариев → playbooks удержания

- **Белые пятна:** удержание через **первый контакт** (первая точка в зоне), а не скидки «всем».
- **Перехват конкурента:** удержание через **сравнимость комиссии/лимитов** в зонах с высокой плотностью конкурента (аналитика, не продукт).
- **Низкая загрузка:** удержание существующих — **качество сервиса точки**, а не расширение сети.

## 6. Модели ML следующего шага (если появятся метки)

- **Классификация «высокий LTV зоны»** по псевдо-меткам (как сейчас для ML-приоритета).
- **Ранжирование зон по ожидаемой марже** после установки АТМ (нужны хотя бы квартальные фактические обороты по точкам).
- **Churn на уровне зоны**, не клиента: падение `total_sum` / `transactions_per_customer` между периодами — тревога для «точки удержания».

## 7. Продуктовые идеи без тяжёлой модели

- В отчёте и на карте: **«топ-5 зон риска»** (конкурент + низкий DS + есть ВТБ) и **«топ-5 возможностей»** (высокий DS, нет ВТБ).
- Экспорт CSV для CRM по **списку H3** с полями скоринга — ручной контур удержания.

---

**Итог:** в рамках текущего проекта удержание логичнее строить как **гео-аналитику и приоритизацию зон/точек**, а не как персональный churn-score; персональные модели потребуют данных, которых нет в статическом `dataset_final.csv`.
